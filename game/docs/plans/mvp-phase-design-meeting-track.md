## 一、已确认的决策基线

- **分层验收**：按「Hub 壳 → 单可用入口 → 关卡竖切 → 带回 → 奖励 → 成长」分层各自可验收推进。
- **数值与细节**：该确定的确定；用最小版本反圈数值与表的工作面，再逐会敲定。
- **Hub 信息量**：主场景可承载内容与显示细节另会专项确认。
- **成长与接口**：升级/升星等该开会开会、该补文档补文档。
- **组队规则**：系统仍为最多 6 人、最少 1 人、自由搭配；MVP 只产 4 名英雄内容，玩家自然最多上场 4 人，不否定 6 人规则。
- **存档**：本地；启动时 5 存档位（选档进入 / 删除）；局内何时存、如何存另会讨论；存档应覆盖全部必要状态含成长。
- **Hub 交互范围**：MVP 仅实现可进入的测试关卡传送门；其余玩法传送门占位可见、不可进，需统一、合适的交互反馈；其他设施交互不做。
- **美术与视觉**：MVP 制作前须单独对齐「引擎方案 + 美术表现方案 + 资源生产方式与验收标准」。不以既往 web 版粗糙效果作为质量基线或默认实现路径；单关 MVP 也需明确目标画面与管线，再进入内容制作。

## 二、后续需讨论敲定的「设计方案总表」（按域分组）

每条建议在会中产出：**结论一句 + 责任文档/表 + 是否进 MVP 边界说明**。

**A. 范围与门禁** — MVP 分层验收标准；二期/不做清单；进入开发阶段的签核条件。

**B. Hub 与传送门** — 最小布局与导航；唯一可用传送门流程；不可用传送门统一交互范式；主场景信息架构与显示细节。

**C. 小队与英雄内容** — 组队规则书面定稿；小队配置入口与存档写入时机；4 英雄定位与技能形态（与战斗 MVP 子集对齐）。

**D. 存档** — 启动流（新档/读档/删档）；序列化数据全集；局内存档机制提案与会审定案。

**E. 测试验证关卡** — 波次与 BOSS；胜负/放弃/重试规则；通关→结算→回 Hub。

**F. 战斗与内容管线（MVP 子集）** — B 系 CSV 最小列、targeting、geometry/windup 子集（registry §2 地城相关收口）；投射物/beam 是否进 MVP。

**G. 数值与经济（MVP）** — 通关奖励；升级；升星与技能解锁边界。

**H. 可读性与 UI 预算** — Hub 同屏信息量、配置界面深度。

**I. 文档与镜像** — 结论回写 `docs/design/`；registry §2 定案项打勾；docs 优先于 client。

**J. 美术与视觉表现（MVP 前必对齐）** — 基线声明（不以 web 旧版为默认标准）；引擎与渲染；角色统一规格；主场景+测试关制作方法；技能与 VFX 分层与 combat 文档对接；管线与分工；单关 MVP 资产表。

- **像素混合策略**：像素不是最终目的，但 MVP 需要能更高成功率、更低成本生产的 2D 像素版资源；同时允许非像素实现（若能更便捷且可保证出素材质量/完整度）。
- **引擎策略**：不先锁死到 Godot；优先评估 Phaser3/phase3 是否能达成视觉与管线需求；若不能，则在新起点重新讨论引擎路线并更新验收口径。
- **AI/工具化美术生产目标**：你给自然语言内容需求，AI/工具链输出“基本可用的美术素材”，并能导入游戏在 MVP 中有效呈现；因此第 2–3 场要把关键资产类型、交付格式、审核口径写进美术总纲与管线文档。

## 三、分场讨论计划（共 11 场）

| 场次 | 主题 | 主要产出 |
|------|------|----------|
| **第 1 场** | MVP 范围、分层验收、二期边界 | 验收清单 + 「不做」列表 + 开发门禁初稿 |
| **第 2 场** | 美术与视觉总纲 | 已定方向：2D像素混合（像素不是核心目的，但MVP优先产出像素版资源；非像素方案可接受但需更便捷/更可保证出素材）；引擎不先锁死，优先评估 Phaser3/phase3 能否达成目标效果；明确美术生产工作流以“自然语言需求→AI/工具生成可用素材→导入并在MVP中有效利用”为目标；web 粗糙版不作基线；单关 MVP 目标画面与性能假设；最小验证方式：doc-only（不做原型），用文档/示例效果评估 |
| **第 3 场** | 美术管线与 MVP 资产规范 | 角色/场景/技能/VFX 制作方法与验收；程序 spec 对接点；MVP 资产清单与分工 |
| **第 4 场** | Hub + 传送门（含禁用反馈） | 交互规范、加载失败策略 |
| **第 5 场** | 小队规则 + 配置流程 + 4 英雄范围 | 规则定稿、UI 流程、英雄/技能形态清单 |
| **第 6 场** | 存档：启动 5 档 + 序列化全集草案 | 字段级清单、删档/空档规则 |
| **第 7 场** | 测试关卡：流程、波次、BOSS、胜负 | 关卡脚本级说明；回填场景与敌人美术需求 |
| **第 8 场** | 战斗与管线 MVP（registry §2 地城子集） | CSV/targeting/geometry/投射物裁剪书面收口 |
| **第 9 场** | 数值：奖励 + 升级 + 升星 | 公式/表结构、与 combat-attributes 对齐 |
| **第 10 场** | 局内存档机制 + Hub 信息架构/UI 预算 | 触发策略定案 + 主场景显示层级规则 |
| **第 11 场** | 总复审与落档 | 开放项清零；含美术结论签核 → 允许进入美术资源效果验证与确认阶段 |

**排期弹性**：第 2+3 场可合并为一场长会（上下半场）；第 10 场 Hub IA 前半在满足条件时可并入第 4 场。

## 五、下一阶段（MVP 开工前置）准入

全部场次结论已落档至约定 spec/表（含美术方向、管线、MVP 资产与验收标准的责任文档），并完成本次签核后，先进入「美术资源效果验证与确认阶段」（以网页演示器中可见效果与层级可读性为验收）；确认通过后再进入「版本内容功能开发」阶段。

美术资源效果验证与确认阶段（建议最小流程）：
- 基于第2-3场定稿的管线制作流程：你提供自然语言需求，AI/工具链输出可用素材草案并导入演示器环境
- 组装演示用技能组合：以 1 个英雄（作为功能模特）为核心，在网页演示器中提供技能列表逐个触发播放：1 个普攻 + 5 个常规技能 + 1 个大招（共 7 个触发项）
- 验收标准：视觉层级与可读性优先（telegraph 可读、特效不遮挡关键读向），并检查粒子/特效不超预算且帧率在目标范围内；不通过则回到资源与配置微调，直至通过

专门讨论会议拆分（满足至少一到两场对齐）：
- 第A场：对齐 1 号英雄的角色模型/基础动画与通用 VFX tag 映射（为本轮技能准备可复用素材框架）
- 第B场：对齐本轮要在演示器里验证的技能效果清单（1 个普攻 + 5 个常规技能 + 1 个大招），形成每个技能的设计方案与美术资源清单并逐项落地
- 第C场：确认美术产出在网页演示器里“能点能播能看”（视觉层级/可读性/帧率），通过后进入版本内容功能开发
