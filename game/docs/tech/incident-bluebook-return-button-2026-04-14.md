# 事件蓝皮书：返回按钮消失事故

**文件编号**: SIXFIGHTER-BLUEBOOK-001
**事件日期**: 2026-04-13 ~ 2026-04-14
**编写日期**: 2026-04-14
**事件等级**: 高（团队十余人次反复修复未果，暴露系统性思维缺陷）
**状态**: 已闭环

---

## 一、事件概述

基地小镇场景中的三个功能测试场景（竞技场、技能演示器、精灵查看器），在从基地切换进入后，右上角的"EXIT"返回按钮始终不可见。

白班期间，团队安排了十余名 AI 员工先后尝试修复。每次修复的思路都集中在按钮本身的属性调整上——调整深度（depth）、调整透明度（alpha）、调整滚动因子（scrollFactor）、强制场景置顶（bringToTop）——每次看似"有道理"，实则全部无效。最终夜班换了一名 AI 员工，在第一轮分析中就精准定位根因并完成修复。

整个事故消耗了十余次完整的"阅读代码→分析→修改→验证"循环，是典型的**高投入、零产出**的调试事故。

---

## 二、技术工程层面分析

### 2.1 根因

在 Phaser 3 的场景插件（ScenePlugin）API 中，`start()` 和 `launch()` 的行为存在关键差异：

| API | 行为 |
|-----|------|
| `this.scene.start(key)` | **先关闭调用者自身的场景**，再启动目标场景 |
| `this.scene.launch(key)` | 只启动目标场景，**不关闭调用者** |

Phaser 3 源码（ScenePlugin.js）：

```javascript
// this.scene.start(key) 的实际执行逻辑
start: function (key, data) {
    this.manager.stop(this.key);   // ← 先停掉自己！
    this.manager.start(key, data); // ← 再启动目标
}

// this.scene.launch(key) 的实际执行逻辑
launch: function (key, data) {
    this.manager.start(key, data); // ← 只启动目标，不停自己
}
```

UIOverlayScene 在 `switchToScene()` 方法中调用了 `this.scene.start('ArenaScene')`。这导致：

1. UIOverlayScene **自身被关闭**（shutdown）
2. ArenaScene 被启动
3. UIOverlayScene 上的所有 UI 元素——返回按钮、摇杆、调试面板——全部随场景销毁而消失
4. 后续代码（`handleSceneSwitch()`、`bringToTop()`、设置按钮可见性）虽然执行了，但场景已在关闭流程中，毫无意义

**一句话总结：不是按钮不可见，是按钮所在的整个场景都不存在了。**

### 2.2 为什么之前的修复全部无效

回顾之前十余次修复中采取的措施：

| 修复手段 | 代码位置 | 为什么无效 |
|----------|---------|-----------|
| `setDepth(9999)` | createReturnButton() | 对象已随场景销毁，depth 再高也无济于事 |
| `setAlpha(1)` | updateUIVisibility() | 不存在的对象没有 alpha 可言 |
| `setScrollFactor(0)` | createReturnButton() | 场景都没了，scrollFactor 失去意义 |
| `scene.bringToTop()` 每帧调用 | update() | update() 随场景关闭而停止运行 |
| `delayedCall(100, bringToTop)` | switchToScene() | 场景已关闭，定时器不会触发 |
| 增加大量 console.log | 遍布全文件 | 日志确实在打印，但没人追问"为什么场景还在执行但对象不显示" |

所有修复的共同特征是：**假定 UIOverlayScene 仍在运行，在此前提下调整按钮的视觉属性**。这个前提本身就是错的。

### 2.3 最终修复方案

修改 `UIOverlayScene.ts` 中两处调用：

```
switchToScene():   this.scene.start(sceneName)      → this.scene.launch(sceneName)
returnToBaseTown(): this.scene.start('BaseTownScene') → this.scene.launch('BaseTownScene')
```

两行代码的改动，解决了十余次修复未能解决的问题。

### 2.4 技术教训

**理解框架 API 的精确语义，而非望文生义。**

`start()` 这个方法名很容易被理解为"启动目标场景"，直觉上不会觉得它会影响到调用者自身。但 Phaser 3 的设计意图是：`start()` 用于场景之间的**替换式切换**（A → B，A 消失），而 `launch()` 用于**并行启动**（A 继续存在，B 也启动）。

这是一个**方法命名与实际行为之间存在认知落差**的典型案例。框架的命名习惯需要通过阅读源码或文档来验证，不能仅凭方法名推断行为。

## 三、团队思维模式层面分析

### 3.1 白班十余人的思维轨迹还原

根据代码中残留的修复痕迹，可以还原出白班团队的大致思维轨迹：

```
第一次尝试：按钮不可见 → 可能是 depth 太低被覆盖了 → setDepth(9999)
    ↓ 失败
第二次尝试：可能需要强制置顶 → bringToTop()
    ↓ 失败
第三次尝试：置顶可能时机不对 → 延迟 100ms 再置顶
    ↓ 失败
第四次尝试：按钮可能被摄像机滚走了 → setScrollFactor(0)
    ↓ 失败
第五次尝试：alpha 可能有问题 → setAlpha(1)
    ↓ 失败
第六次尝试：每帧都 bringToTop 总行了吧 → 写进 update()
    ↓ 失败
……
第N次尝试：加更多日志看看发生了什么 → 大量 console.log
    ↓ 失败（日志都正常打印，但按钮就是不出现）
```

这是一个典型的**逐层加码式调试**：每一轮都在上一轮的基础上叠加新措施，但从未退后一步重新审视问题本身。

### 3.2 问题出在哪里：三层思维缺陷

#### 缺陷一：在最表层寻找原因（调试叶子，而非调试根）

十余人不约而同地将注意力集中在"按钮为什么不可见"这个最表层的问题上。他们阅读了 `createReturnButton()`、`updateUIVisibility()`、`switchUIMode()` 这些方法，看到按钮的创建逻辑正确、可见性切换逻辑正确，于是认定问题出在视觉渲染层面。

没有人问出那个致命的问题：**"UIOverlayScene 这个场景本身，在切换到 ArenaScene 之后，还存在吗？"**

这是一个**自上而下**的调试习惯——从最接近用户可见症状的地方开始排查——而不是**自下而上**的调试习惯——从系统的基础运行时状态开始排查。正确的做法是先验证基础设施（场景是否存活），再验证上层逻辑（按钮是否可见）。

#### 缺陷二：将"不工作"等同于"属性不对"

整个白班的假设都是"按钮存在但属性不对"。这是一个**隐含前提**，所有人都没有意识到自己在做这个假设。

如果有人停下来做一次前提审查："我们是在假定场景还存在——这个假定成立吗？"问题就会立刻暴露。

**思维方式升级：在调试任何"不工作"的问题时，先验证三个前提：**
1. **存在性前提**：被操作的对象是否存在？
2. **可达性前提**：代码路径是否真的被执行到了？
3. **正确性前提**：执行时的状态是否符合预期？

这三个前提按优先级递减。存在性是最优先的——如果对象都不存在了，讨论它的属性毫无意义。

#### 缺陷三：缺乏"杀掉所有假设重来"的意识

在第一次到第五次修复都无效后，合理的做法是**推翻之前的所有假设，重新理解系统**。但十余人接力式地在前人的修复基础上继续叠加，没有一个人说"停下来，前面所有尝试的前提可能就是错的"。

这是一种**路径依赖**思维：前人的工作（哪怕已证明无效）成为了后人的思维起点，后人不自觉地在前人划定的范围内寻找答案。

**防范措施：设立"三次法则"——如果同一方向连续尝试三次仍未解决，必须停止当前方向，重新进行基础性分析。**

### 3.3 为什么夜班一个人就能解决

夜班这名员工的做法与白班有本质区别：

**白班的思维路径：**
```
看到"按钮不可见"
→ 阅读 UIOverlayScene 中按钮相关代码
→ 尝试调整按钮属性
→ 失败，叠加更多调整
→ 继续失败……
```

**夜班的思维路径：**
```
看到"按钮不可见"
→ 先通读整个场景系统的代码（UIOverlayScene + BaseTownScene + ArenaScene + BootScene + main.ts）
→ 理解场景启动流程：BootScene.start() → BaseTownScene + UIOverlayScene.launch()
→ 理解场景切换流程：UIOverlayScene.start() → ???
→ 发现矛盾点：BootScene 用 launch() 启动 UIOverlayScene，但 UIOverlayScene 自己切换场景时用了 start()
→ 查证 Phaser 3 ScenePlugin 源码中 start() 的精确语义
→ 确认 start() 会关闭调用者自身
→ 根因定位，修复完成
```

核心差异在于：**夜班员工先理解了系统是怎么运作的，再去找问题出在哪里；白班员工是先看到症状，然后在症状附近翻找原因。**

### 3.4 对团队培训的启示

#### 启示一：先读系统，再读症状

当一个问题涉及多个模块交互时（本例中是两个场景的协同），必须先完整理解交互机制，再定位具体问题。不要在看到症状后就立刻钻进最近的代码里。

实操建议：遇到跨模块 Bug 时，强制要求先画出**模块交互流程图**，标注每一步的数据流向和生命周期。在流程图上标出"症状出现在哪一步"，然后从那一步**向上游**追溯。

#### 启示二：对框架 API 保持"不信任"态度

永远不要假设自己理解了一个框架方法的确切行为，特别是当方法名看起来"很直观"时。`start()` 这个名字太直觉了，以至于没人觉得需要去查文档或源码。

实操建议：在使用任何框架 API 之前，特别是涉及**生命周期管理**的 API，必须阅读其文档中的完整行为描述，而非仅凭方法名推断。将"查证 API 精确语义"作为调试流程的必要步骤。

#### 启示三：学会问"最蠢的问题"

白班十余人都没有问出"场景还存在吗？"这个问题，因为它看起来太蠢了——场景当然存在啊，代码都写在场景里面，日志都在打印。

但在软件系统中，最违反直觉的事实往往就是真相。学会问那些"看起来很蠢"的基础性问题，是高级调试能力的核心。

实操建议：在调试陷入僵局时，列出你对系统的所有"理所当然"的假设，逐一验证。特别关注那些"太明显了所以没人会质疑"的假设——它们是最可能出问题的地方。

#### 启示四：失败是有惯性的

白班十余人的修复痕迹留在了代码中（高 depth、强制 alpha、每帧 bringToTop 等）。这些痕迹向后来者传递了一个隐含信号："前人已经在这个方向上做了大量工作，问题应该在这个方向上"。这形成了**失败惯性**——前人失败的方向，反而因为修复痕迹的存在，变成了后人最容易进入的方向。

实操建议：当一次修复尝试失败后，**回滚修改**，不要在代码中保留无效的修复痕迹。干净的代码让后来者不会被误导。

## 四、防范措施总结

| 编号 | 措施 | 适用场景 |
|------|------|---------|
| P-001 | 跨模块 Bug 先画交互流程图，从症状向上游追溯 | 所有涉及多模块交互的 Bug |
| P-002 | 调试前先验证三个前提：存在性→可达性→正确性 | 所有"不工作"类 Bug |
| P-003 | 使用框架 API 前查证其精确语义，不凭方法名推断 | 所有涉及框架生命周期 API 的代码 |
| P-004 | 同一方向连续三次失败后，必须停止并重新分析 | 所有调试过程 |
| P-005 | 失败的修复必须回滚，不在代码中保留无效痕迹 | 所有代码修改 |
| P-006 | 调试僵局时列出并验证所有"理所当然"的假设 | 复杂 Bug 调试 |

## 五、结论

这次事故不是技术能力问题，而是**思维方法问题**。十余人并非不知道 `start()` 和 `launch()` 的区别——他们很可能根本没想到要去考虑这个问题，因为他们的注意力被锁定在了"按钮属性"这个更浅的层面上。

**一句话总结这次教训：当你调试一个"不工作"的问题时，最该做的不是问"它为什么不工作"，而是先问"它还在不在"。**
