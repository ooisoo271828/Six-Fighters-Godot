## 1. 目的与适用范围

### 1.1 目的

本手册定义技能开发的**标准化流程**，确保：
- 每个技能都有完整、可追溯的设计文档
- VibeCoding 工作流（自然语言 → AI → 产出）有章可循
- 多人/多机协作时不产生混乱
- 技能视觉效果与游戏整体视觉规范一致

### 1.2 适用范围

- 所有战斗技能的设计与实现
- 技能视觉特效（VFX）的开发
- 技能相关的参数调优与迭代

## 3. Phase 1：技能设计文档

### 3.1 创建文档

1. 复制 `docs/design/visual-rules/skills/skill_design_template.md`
2. 按命名规范重命名（参见 `skills/README.md`）
3. 填写元数据，Status 设为 `Draft`
4. 与 AI 助手讨论需求，逐步填写各章节

### 3.2 设计讨论的输入输出

**人类输入**（自然语言）：
```
"我需要一个火球术技能，发射类，有红色火球飞行，
命中后爆炸，爆炸范围中等，视觉效果要华丽。"
```

**AI 助手输出**：
- 结构化的设计文档草稿
- 必要时提供视觉参考建议
- 确认技术可行性

### 3.3 设计文档评审 Checklist

- [ ] 所有章节已填写（或明确标注"不适用"）
- [ ] 视觉描述足够具体，AI 可以据此生成实现
- [ ] 参数配置章节填写了关键数值
- [ ] 可读性约束已考虑
- [ ] 与 `pixel-art-visual-bible.md` 不冲突

评审通过后，Status 改为 `待实现`。

## 4. Phase 2：程序化生成（混合方案 C）

### 4.1 核心架构原则

**数值与视觉分离**：

```
┌─────────────────────────────────────────────────────────────┐
│                 技能数值（CSV 表格）                         │
│  docs/design/combat-rules/values/skill-values.csv          │
│-----------------------------------------------------------  │
│  - base_damage（基础伤害）                                  │
│  - cooldown（冷却时间）                                     │
│  - cast_range（施法距离）                                   │
│  - cast_time（施法时间）                                    │
│  - travel_ms（飞行/持续时长）                               │
│  - telegraph_ms（预警时长）                                 │
│  - damage_type（伤害类型）                                  │
│  - impact_level（命中等级）                                 │
│  - behavior 相关参数（projectile_count, max_bounces 等）  │
└─────────────────────────────────────────────────────────────┘
                        ↓ 读取
┌─────────────────────────────────────────────────────────────┐
│                SkillDef.gd / SkillDef.tres                  │
│  从 CSV 加载数值，提供给游戏逻辑                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│               技能视觉（.tres + .tscn）                      │
│  SkillVisualDef.tres（视觉参数）                             │
│  vfx_{skill_id}.tscn（粒子系统、动画）                       │
│-----------------------------------------------------------  │
│  - 粒子系统参数（amount, lifetime, color_gradient 等）     │
│  - 动画参数（时长、hit_stop 帧数等）                        │
│  - 颜色方案（主体色、高光色、暗部色等）                     │
│  - 速度、大小、缩放曲线等视觉参数                           │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 方案说明

**混合方案 C** 是本项目的标准实现方式：

```
代码生成结构（GDScript）         .tres 配置参数（视觉）
─────────────────────    ─────────────────────────────
- 粒子系统节点层级          - 粒子数量、生命周期
- 材质创建逻辑              - 颜色渐变
- 动画播放逻辑              - 大小变化曲线
- 脚本组件挂载              - 速度、扩散角度等（视觉）

CSV 数值表格（战斗数值）
─────────────────────
- 伤害、冷却、施法距离等
- 由 SkillDef 在运行时加载
```

**为什么这样设计**：
- 数值与视觉彻底解耦，方便数值策划调整数值，不影响视觉
- 结构与参数分离，修改参数不需要改代码
- `.tres` 文件可在 Godot 编辑器中可视化调整
- 代码生成保证一致性和可复用性

### 4.3 与现有系统集成

技能实现需要生成/修改以下文件：

| 文件 | 说明 | 生成方式 |
|------|------|---------|
| `skill-values.csv` | 技能数值（伤害、冷却等） | 手动编辑或程序化生成行 |
| `SkillDef.tres` | 技能定义（引用 CSV 或包含基础配置） | 程序化生成 |
| `SkillVisualDef.tres` | 视觉定义（VFX 场景引用、视觉参数） | 程序化生成 |
| `vfx_{skill_id}.tscn` | VFX 场景（粒子系统、动画） | 代码生成结构 + .tres 配置 |
| `projectile_{skill_id}.tscn` | 投射物场景（发射类技能） | 代码生成 |

这些文件统一注册到 `SkillRegistry`，`skill_demo` 场景会自动加载。

### 4.3 VibeCoding 工作流：自然语言 → 实现

**人类输入**（自然语言）：
```
"根据 001_proj_fireball.md 设计文档，
生成火球术的完整实现，包括 SkillDef、
SkillVisualDef 和 VFX 场景。"
```

**AI 助手执行步骤**：
1. 读取设计文档（在 `docs/design/visual-rules/skills/` 目录）
2. **读取数值配置**：从 `docs/design/combat-rules/values/skill-values.csv` 中过滤 `category == "{skill_id}"` 的行，获取所有数值参数
3. 生成/修改对应的 `.tres` 资源文件（SkillDef.tres, SkillVisualDef.tres）
4. 生成 VFX 场景（`.tscn`）
5. 确认文件写入项目正确位置
6. 报告完成，等待验证

**数值与视觉的分工**：
- **CSV 数值表格**：包含所有战斗数值（伤害、冷却、施法距离、timing 等）
  - AI 读取 CSV 并应用到 SkillDef
  - 数值策划可以直接编辑 CSV，不需要修改 .tres 或 .tscn
- **.tres 视觉参数**：包含视觉相关参数（粒子系统、颜色、动画等）
  - AI 根据设计文档的「视觉设计」章节生成
  - 可以在 Godot 编辑器中手动微调

**手动调整**（极少情况）：
- 如果 AI 生成的特效不令人满意，人类可以在 Godot 编辑器中打开 `.tscn` 或 `.tres` 手动微调
- 手动调整后，必须将调整内容**回写到设计文档的参数配置章节**

## 5. Phase 3：验证（使用 skill_demo）

### 5.1 skill_demo 场景功能说明

`scenes/dev/skill_demo.tscn` 是技能验证的专用场景。

**功能**：
- 从 `SkillRegistry` 动态加载所有已注册技能
- 下拉菜单选择要测试的技能
- 多种靶标阵型：单目标 / 双目标 / 三角阵 / 散开群 / 一字排
- 播放控制：▶ 播放 / ⏸ 暂停 / ⏹ 停止
- 速度调节：0.5× / 1× / 2×（方便观察细节）
- 循环播放：开启后可重复施放，方便观察重复性

### 5.2 验证操作步骤

1. 在 Godot 编辑器中打开 `skill_demo.tscn`
2. 运行场景（F5 或点击播放）
3. 在下拉菜单中选择要验证的技能
4. 观察技能表现：
   - 施法前摇是否流畅
   - 飞行/爆发效果是否符合设计
   - 颜色、大小、速度是否合适
5. 切换靶标阵型，验证多目标下的表现
6. 调节速度，观察细节
7. 开启循环，验证重复性

### 5.3 验证不通过的处理

1. **用自然语言向 AI 助手描述问题**：
   ```
   "火球术的爆炸效果太小了，而且颜色偏橙，
   应该是亮黄色。爆发半径也感觉不够大。"
   ```

2. **AI 助手根据反馈修改**：
   - 修改对应的 `.tres` 参数文件
   - 或调整 VFX 场景中的节点参数
   - 更新设计文档的**参数配置**章节

3. **重新验证**，直到通过

## 6. Phase 4：迭代循环规范

### 6.1 触发迭代的情况

- 验证不通过
- 设计需求变更
- 平衡性调整
- 视觉效果优化

### 6.2 迭代步骤

```
1. 更新设计文档
   - 修改对应的章节
   - 在「迭代历史」表格中记录变更

2. 重新生成实现
   - AI 助手根据更新后的文档重新生成/修改文件

3. 重新验证
   - 在 skill_demo 中验证修改效果

4. 定稿
   - 验证通过后，将设计文档 Status 改为 `Verified`
```

### 6.3 重要原则

> **始终以设计文档为唯一真理源（Single Source of Truth）。**
> 任何对技能实现的修改，都必须先更新设计文档。

## 7. 多机协作注意事项

### 7.1 路径规范

- **禁止**在文档或代码中使用绝对路径
- **必须**使用 Godot 的 `res://` 相对路径
- 示例：
  - ✅ `res://skills/defs/fireball.tres`
  - ❌ `D:\Vibe Coding\Six-Fighters-Godot\game\skills\defs\fireball.tres`

### 7.2 Git 规范

- 技能设计文档（`.md`）必须提交到版本控制
- 生成的 `.tres` 和 `.tscn` 文件也必须提交
- 每次完整的技能实现（Phase 1-3）应该是一个独立的 commit

### 7.3 状态同步

- 切换机器前，确保设计文档 Status 字段是最新的
- 如果实现未完成，在文档中注明当前进度

## 8. AI 助手使用指南

### 8.1 提供给 AI 的上下文

每次开始技能开发任务时，应该向 AI 助手提供：

1. 对应的技能设计文档路径
2. 本文档（工作手册）路径
3. `pixel-art-visual-bible.md` 路径（视觉效果约束）
4. **技能数值 CSV 表格路径**（数值配置）
5. 现有类似技能的实现作为参考（如有）

示例：
```
请参考以下文档，实现 001_proj_fireball.md 中描述的技能：
- 工作手册：docs/design/skill-development-workflow.md
- 设计文档：docs/design/visual-rules/skills/001_proj_fireball.md
- 数值配置：docs/design/combat-rules/values/skill-values.csv
  - 查找方式：过滤 category == "fireball_basic"
- 视觉规范：docs/design/visual-rules/pixel-art-visual-bible.md
- 参考实现：skills/defs/ice_arrow.tres（如有）
```

### 8.2 AI 读取数值 CSV 的规范

AI 助手在处理技能实现时，必须：

1. **读取 CSV 数值表格**：
   - 文件路径：`docs/design/combat-rules/values/skill-values.csv`
   - 过滤条件：`category == "{skill_id}"`（skill_id 来自设计文档）

2. **应用到 SkillDef**：
   - 将 CSV 中的数值参数（base_damage, cooldown, cast_range 等）应用到 SkillDef.tres
   - 或确保在运行时 SkillDef 可以从 CSV 加载这些数值

3. **视觉参数在 .tres 中**：
   - 粒子系统参数、颜色方案、动画参数等视觉相关参数在 SkillVisualDef.tres 中
   - 这些参数由设计文档的「视觉设计」和「参数配置」章节定义

### 8.3 AI 输出规范

- 写入文件前，确认目标路径存在
- 生成 `.tres` 文件时，严格遵循 Godot 资源文件格式规范
  - `[resource]` 块下属性必须有 Tab 缩进
  - 行尾用 CRLF（`\r\n`）
- 生成 `.tscn` 文件时，严格遵循 Godot 场景文件格式规范
  - `[node]` 块下属性行零缩进
  - `[resource]` 块下属性行必须有 Tab 缩进
- 完成后，报告所有创建/修改的文件路径
- **重要**：确保数值参数来自 CSV，不要在 .tres 中硬编码数值

## 9. 常见问题（FAQ）

### Q1：设计文档要不要版本控制？

**要。** 设计文档是项目资产的一部分，必须提交到 Git。

### Q2：如果 AI 生成的特效完全不对怎么办？

先检查设计文档是否描述得足够具体。如果文档没问题，可以：
1. 在 Godot 编辑器中手动创建一个理想效果的粒子系统
2. 将其参数反写到设计文档的「参数配置」章节
3. 让 AI 基于更新后的参数重新生成

### Q3：技能太多了，管理不过来怎么办？

- 用设计文档的 `设计优先级` 字段管理开发顺序
- 定期在 `skills/README.md` 中维护技能清单表格
- 已验证的技能和草稿状态的技能分开管理

### Q4：数值参数应该放在哪里？

**必须放在 CSV 数值表格中**：`docs/design/combat-rules/values/skill-values.csv`

原因：
- 数值策划可以直接编辑 CSV，不需要修改 .tres 或 .tscn
- 数值与视觉彻底解耦，修改数值不会影响视觉效果
- 方便批量调整和平衡性测试

### Q5：设计文档中的「战斗参数」表格还需要吗？

**不需要。** 设计文档只关注「视觉表现 + 功能逻辑」，数值部分指向 CSV 表格即可。

在设计文档的「基本信息」章节中：
- 伤害类型：填写 `*见 CSV 数值表*`
- 添加一个引用说明，指向 `docs/design/combat-rules/values/skill-values.csv`

### Q6：旧的 .tres 文件（包含数值字段）怎么处理？

- 如果 `.tres` 文件是通过程序化生成的（在 `resources/skills/` 目录），可以安全删除
- 设计文档应该引用新的数值 CSV + 视觉 .tres
- 旧的、手动创建的 `.tres` 文件应该标记为 `Deprecated` 或删除
